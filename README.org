#+TITLE: Datum
#+AUTHOR: Sebastian Monia & Gary Hollis
#+OPTIONS: toc:3 num:nil

An Emacs SQL IDE backed by a Python CLI.  Connect to any ODBC database,
get instant completion, goto-definition, parameter signatures, one-keystroke
table operations, and import/export -- all without leaving your editor.

Datum keeps all compute in Python and uses Emacs as a smart terminal.
Structured data flows between them via an envelope protocol that is
invisible to the user but keeps the Emacs side informed about dialect,
metadata, and introspection results.  The Python CLI is also usable
standalone from any terminal.

* Features

- Connect to any database with an ODBC driver (MSSQL, PostgreSQL, SQLite, MySQL, Oracle, ...)
- Auto-detect SQL dialect or set it explicitly
- Introspection commands: =:tables=, =:columns=, =:schemas=, =:databases=, =:routines=, =:running=
- Auto-introspect on connect: completion is populated automatically (no manual =:tables= needed)
- Export query results to CSV, Parquet, or JSON (=:out=)
- Import files into SQL tables with schema inference (=:in=)
- Custom query templates with placeholders and ODBC parameters
- Native org-mode table output: query results are valid org tables, ready to yank into any =.org= document
- Emacs integration: completion-at-point, eldoc parameter signatures, mode line status, keybindings
- Optional Polars backend for faster import/export (falls back to pyarrow or stdlib)

* Installation

** Python package

#+begin_src shell
pip install git+https://github.com/ghollisjr/datum.git
#+end_src

Or for development (changes take effect immediately):

#+begin_src shell
cd /path/to/datum
pip install -e .
#+end_src

* Connecting to a database

** Using individual parameters

#+begin_src shell
datum --driver "ODBC Driver 18 for SQL Server" \
      --server myserver --database mydb \
      --user sa --pass ENV=DB_SECRET
#+end_src

** Using a DSN

#+begin_src shell
datum --dsn MyDSN
#+end_src

** Using a connection string

#+begin_src shell
datum --conn-string="Driver={PostgreSQL Unicode};Server=localhost;Database=mydb;Uid=postgres;Pwd=secret"
#+end_src

** Environment variable substitution

Any parameter value starting with =ENV== pulls from an environment variable:

#+begin_src shell
datum --driver "PostgreSQL Unicode" --server localhost \
      --user ENV=PG_USER --pass ENV=PG_PASS --database mydb
#+end_src

** Dialect detection

Datum auto-detects the SQL dialect from the driver name or DSN.  You can
override it with =--sql-type=:

#+begin_src shell
datum --dsn MyDSN --sql-type postgres
#+end_src

Supported dialects: =mssql=, =postgres=, =mysql=, =sqlite=, =oracle=, =ansi= (fallback).

* Query input

Type SQL at the =>= prompt.  Queries are sent to the server when you:

- End a line with =;;=
- Type =GO= on its own line

#+begin_example
>select * from users;;

>select *
>from orders
>where total > 100
>GO
#+end_example

** Parameterized queries

Use =?= as an ODBC parameter placeholder.  Datum prompts for each value:

#+begin_example
>select * from users where name = ? and age > ?;;
1>Alice
2>30
#+end_example

Values are properly escaped by pyodbc.

* Output format

All query results and introspection output are formatted as org-mode
tables:

#+begin_example
| id | name  | email             | created_at          |
|----+-------+-------------------+---------------------|
|  1 | Alice | alice@example.com | 2024-01-15 09:30:00 |
|  2 | Bob   | bob@example.com   | 2024-01-16 14:22:00 |

Rows printed: 2/2
#+end_example

This means every result you see -- queries, =:tables=, =:columns=,
=:running=, everything -- is a valid org table.  You can yank results
directly into any =.org= document and they align with =TAB=
immediately, no reformatting needed.

Use =C-c s y= to copy the last result to the kill ring, ready to paste.

* Built-in commands

All commands start with =:= and are entered at the prompt.

| Command             | Description                                    |
|---------------------+------------------------------------------------|
| =:databases [pattern]= | List databases (filter with SQL LIKE wildcards) |

** Export (=:out=)

Export the *next* query's results to a file.  Format is inferred from the
file extension.

#+begin_example
>:out /tmp/results.csv
:out target set to: /tmp/results.csv (will error if file exists)
Run your query now. The next query's results will be exported.

>select * from orders;;
#+end_example

Supported formats: =.csv=, =.parquet=, =.json= (newline-delimited).

When a query produces duplicate column names (e.g. =SELECT a.id, b.id
FROM ...=), they are automatically disambiguated with =_1=, =_2= suffixes,
and a warning is shown.

Use =:force= to overwrite an existing file:

#+begin_example
>:out /tmp/results.parquet :force
#+end_example

** Import (=:in=)

Import a file into a SQL table.  Schema is inferred automatically.

#+begin_example
>:in /tmp/data.csv my_table
#+end_example

Modes:

| Syntax                                   | Behavior                      |
|------------------------------------------+-------------------------------|
| =:in file table=                         | Error if table exists         |
| =:in file table :insert=                | Append to existing table      |
| =:in file table :replace=               | Drop, recreate, and insert    |
| =:in file table :batch 500=             | Custom batch size (default 1000) |

Supported formats: =.csv=, =.parquet=, =.json=.

The =:batch N= flag controls how many rows are sent per =executemany= call.
Lower values use less memory (useful for wide tables); higher values are faster.
The default is 1000.

When Polars is installed, it handles file reading and type inference.
Otherwise falls back to pyarrow (for parquet/json) or stdlib csv.

** Legacy CSV export

#+begin_example
>:csv /tmp/session_output.csv
#+end_example

Appends all subsequent query output to the file until disabled with =:csv=
(no argument).

** Script execution

#+begin_example
>:script /path/to/query.sql
#+end_example

Reads and executes a SQL file.  Supports ={placeholder}= and =?= parameters
like custom commands.

* Configuration file

Datum looks for an INI file in this order:

1. The path given by =--config=
2. =$XDG_CONFIG_HOME/datum/config.ini=
3. =~/.config/datum/config.ini=

** Example configuration

#+begin_src ini
[general]
rows_to_print=0
column_display_length=0
null_string=[NULL]
newline_replacement=[NL]
tab_replacement=[TAB]

[queries]
top=SELECT * FROM {table} LIMIT {how_many}
find=SELECT * FROM {table} WHERE {column} LIKE '%{pattern}%'
#+end_src

** Custom commands

Queries in the =[queries]= section become commands prefixed with =:=.
They support two levels of parameterization:

- ={placeholder}= -- replaced via Python string formatting (prompted interactively)
- =?= -- ODBC parameterized values (escaped by pyodbc)

#+begin_example
>:top
table>users
how_many>10

Command query:
 SELECT * FROM users LIMIT 10
#+end_example

You can combine both:

#+begin_src ini
[queries]
search=SELECT * FROM {table} WHERE {field} = ?
#+end_src

#+begin_example
>:search
field>name
table>users

Command query:
 SELECT * FROM users WHERE name = ?
1>Alice
#+end_example

* Emacs integration

** Setup

#+begin_src emacs-lisp
(use-package sql-datum
  :load-path "/path/to/datum"
  :after sql
  :config
  ;; PostgreSQL via DSN
  (add-to-list 'sql-connection-alist
               '("my-postgres"
                 (sql-product 'datum)
                 (sql-send-terminator ";;")
                 (sql-server "localhost")
                 (sql-database "mydb")
                 (sql-user "postgres")
                 (sql-password 'ask)
                 (sql-datum-options '("--driver" "PostgreSQL Unicode"))))
  ;; MSSQL with integrated auth
  (add-to-list 'sql-connection-alist
               '("my-mssql"
                 (sql-product 'datum)
                 (sql-send-terminator ";;")
                 (sql-server "sqlserver.local")
                 (sql-database "northwind")
                 (sql-user "")
                 (sql-password "")
                 (sql-datum-options '("--driver" "ODBC Driver 18 for SQL Server"
                                      "--integrated"))))
  ;; SQLite
  (add-to-list 'sql-connection-alist
               '("my-sqlite"
                 (sql-product 'datum)
                 (sql-send-terminator ";;")
                 (sql-server "")
                 (sql-user "")
                 (sql-password "")
                 (sql-datum-options '("--driver" "SQLite3"))
                 (sql-database "/path/to/database.db"))))
#+end_src

Then use =M-x sql-connect= and pick a connection, or =M-x sql-datum= to be
prompted for parameters interactively.

** Password handling

Three options for =sql-password=:

- A literal string (visible in =list-processes=)
- ='ask= -- prompted with =read-passwd= each time
- ='auth-source= -- looked up via Emacs =auth-source= (supports =.authinfo.gpg=, GNOME Keyring, etc.)

For =auth-source=, add an entry keyed by connection name:

#+begin_example
machine my-postgres login postgres password secret123
#+end_example

By default, passwords are passed via a temporary environment variable
(=SQLDATUMPASS=) so they don't appear in process listings.

** Connection gotchas

*** Always set all login fields explicitly

Emacs =sql-mode= stores =sql-server=, =sql-database=, =sql-user=, and
=sql-password= as global variables.  When you connect, these are set to
whatever that connection entry specifies — but fields you *omit* keep
their previous value.  This means a password from connection A can leak
into connection B if B doesn't explicitly clear it.

*Rule of thumb:* every entry in =sql-connection-alist= should set all
four fields, even if the value is an empty string.

#+begin_src emacs-lisp
;; BAD — if a previous connection set sql-password, it leaks here:
'("my-local-pg"
  (sql-product 'datum)
  (sql-server "")
  (sql-user "")
  ;; sql-password not set — inherits from last connection!
  (sql-datum-options '("--conn-string=Driver={PostgreSQL Unicode};...")))

;; GOOD — explicitly clear the password for peer/ident auth:
'("my-local-pg"
  (sql-product 'datum)
  (sql-server "")
  (sql-database "")
  (sql-user "")
  (sql-password "")
  (sql-datum-options '("--conn-string=Driver={PostgreSQL Unicode};...")))
#+end_src

This is especially important when switching between password-authenticated
connections (e.g. Docker test databases) and passwordless connections (e.g.
local PostgreSQL with peer auth).

** Keybindings

All keybindings are active in =sql-mode= buffers.

*** Table operations (=C-c t=)

| Key       | Command                  | Description                                  |
|-----------+--------------------------+----------------------------------------------|
| =C-c t e= | =sql-datum-export=       | Export table to file (=C-u= for custom query)   |
| =C-c t i= | =sql-datum-import=       | Import file into table (=C-u= for batch size)   |
| =C-c t t= | =sql-datum-top=          | Select top N rows (default 10, =C-u N=)      |
| =C-c t c= | =sql-datum-count=        | Count rows (=COUNT_BIG= on MSSQL)            |
| =C-c t s= | =sql-datum-sample=       | Random sample of N rows (default 10, =C-u N=) |
| =C-c t d= | =sql-datum-describe=     | Describe table columns (alias for =columns=) |
| =C-c t x= | =sql-datum-table-exists= | Check if table exists                        |
| =C-c t D= | =sql-datum-drop-table=   | Drop table (with confirmation)               |

All table operations default to the identifier at point when it matches a
known table name, so you can place your cursor on a table name and press
the keybinding without re-typing it.

The quick-query commands (=top=, =sample=, =count=, =table-exists=) are
dialect-aware: they use =TOP= / =NEWID()= / =OBJECT_ID()= on MSSQL and
=LIMIT= / =RANDOM()= / =to_regclass()= on PostgreSQL and other dialects.

| =C-c s t= | =sql-datum-tables=       | List tables (=C-u= to filter with LIKE)                    |
| =C-c s c= | =sql-datum-columns=      | List columns (prompts for table)                           |
| =C-c s d= | =sql-datum-databases=    | List databases (=C-u= to filter with LIKE)                 |
| =C-c s s= | =sql-datum-schemas=      | List schemas (=C-u= to filter with LIKE)                   |
| =C-c s r= | =sql-datum-running=      | Show running queries                                       |
| =C-c s R= | =sql-datum-routines=     | List routines (=C-u= to filter with LIKE)                  |
| =C-c s v= | =sql-datum-version=      | Show server version                                        |
| =C-c s u= | =sql-datum-user=         | Show current user                                          |
| =C-c s f= | =sql-datum-refresh=      | Refresh introspection data (including cross-db caches)     |
| =C-c s y= | =sql-datum-copy-last-result= | Copy last result to kill ring (already org table format)    |
| =C-c s F= | =sql-datum-toggle-auto-refresh= | Toggle periodic introspection refresh (=C-u N= for interval) |

*** Connection management (=C-c C-x=, SLIME-style)

| Key             | Command                  | Description                          |
|-----------------+--------------------------+--------------------------------------|
| =C-c C-x c=     | =sql-connect=            | Connect to a named connection        |
| =C-c C-x n=     | =sql-set-sqli-buffer=    | Switch active SQLi buffer            |
| =C-c C-x s=     | =sql-datum-scratch=      | Open scratch SQL buffer (=C-u= for additional) |
| =C-c C-x d=     | =sql-datum-disconnect=   | Disconnect session (scratch buffers kept)        |

The final key in each sequence also works with Control held (e.g.
=C-c C-x C-c= is equivalent to =C-c C-x c=), following the SLIME
convention.

** Scratch buffer

The scratch buffer is a convenient, fast way to write and send SQL
interactively.  =sql-datum-scratch= opens (or switches to) a
=*datum-scratch*= buffer in =sql-mode=, wired to the active datum
connection.  Unlike the SQLi REPL, a scratch buffer gives you a full
editing environment: multi-line queries, undo history, =M-.=
goto-definition, completion, and all the =C-c t= / =C-c s=
keybindings.  Use =C-c C-r= to send a region or =C-c C-b= to send the
whole buffer.

With a prefix argument (=C-u=), it creates an additional scratch buffer
with a numeric suffix -- useful for keeping separate workspaces (one for
exploration, one for DDL, one for a long analysis).

Note that all datum features -- completion, =M-.= goto-definition,
eldoc, keybindings -- work in any =sql-mode= buffer associated with a
datum connection, not just scratch buffers.  So =.sql= files work just
as well; the scratch buffer is simply a fast way to start writing SQL
without creating a file.

A typical workflow:

1. =M-x sql-connect= -- connect to a database
2. =C-c q= -- open the scratch buffer (completion is already populated)
3. Write SQL, =C-c C-c= to run region or paragraph, =M-.= to inspect objects, iterate

Bind it globally for quick access:

#+begin_src emacs-lisp
(autoload 'sql-datum-scratch "sql-datum" nil t)
(global-set-key (kbd "C-c q") #'sql-datum-scratch)
#+end_src

** Completion at point

Table, schema, column, and routine names are available for completion in
both the SQLi buffer and =sql-mode= editing buffers.  Works with
=completion-at-point= (=C-M-i=), Company, Corfu, etc.

When =sql-datum-auto-introspect= is non-nil (the default), completion data
is populated automatically on connect — no need to run =:tables= or
=:routines= manually.  You can also run =:refresh= at any time to silently
update the completion data.

*** Contextual column completion

Column completion is context-aware: when the current SQL statement
references specific tables (via =FROM=, =JOIN=, =UPDATE=, or =INTO=),
only columns from those tables are offered.  The scanner reads the full
statement -- both before and after point -- so =SELECT c<TAB> FROM
customers= correctly narrows to the =customers= columns.

Columns are fetched lazily.  When you =TAB= and the columns for a
referenced table haven't been cached yet, Datum silently sends a
=:columns= request in the background.  The first =TAB= falls back to all
known columns; the second =TAB= (once the fetch completes) shows the
narrowed contextual set.  Tables that were previously introspected with
=:columns= are used from cache immediately.

Multi-table statements work as expected: =FROM t1 JOIN t2 ON ... WHERE
<TAB>= offers columns from both =t1= and =t2=.  When no tables can be
detected (e.g. typing =SELECT <TAB>= before a =FROM= clause has been
written), completion falls back to all cached columns.

When contextual columns are available, they are sorted before tables,
schemas, and other object types in the completion list.

*Note:* Table aliases are not resolved in the current version.
=FROM users u WHERE u.<TAB>= will not contextually filter -- it falls back
to all columns.

*** Cross-database completion (MSSQL)

On MSSQL, completion supports three-part names (=other_db.schema.table=).
Type a database name, complete it with =TAB=, type =.=, and hit =TAB=
again -- Datum fetches that database's tables, routines, and schemas
asynchronously on demand.  The first =TAB= after a new database prefix
triggers a background fetch; press =TAB= again once the fetch completes
to see completions.  Results are cached per-session so subsequent
completions are instant.

Object names containing dots are bracket-quoted automatically (e.g.
=master.dbo.[My.Dotted.Name]=) so they display as valid three-part
references and don't get confused with additional qualifiers.

Running =C-c s f= (=sql-datum-refresh=) refreshes not only the current
database but also re-fetches any cross-database caches built during the
session, so dropped or newly created objects are picked up everywhere.

This works because MSSQL supports cross-database queries via three-part
catalog references.  PostgreSQL cannot query across databases without
reconnecting, so this feature is MSSQL-only.

You can also introspect a remote database manually with =:refresh-db <name>=.

*** Quoted identifier completion

Identifiers containing spaces, hyphens, or other special characters are
automatically quoted when completed.  PostgreSQL uses double quotes
(="my column"=), MSSQL uses square brackets (=[my column]=).  Only the
segments that need quoting are wrapped — e.g. =public."my table"=.

You can start typing with an opening quote or bracket (="che<TAB>= or
=[che<TAB>=) and completion will match against the bare name, then
produce the correctly quoted result without double-quoting.

*** Schema-qualified completion

Completion supports schema-qualified names.  Both bare names (=users=) and
schema-qualified names (=dbo.users=) are available as candidates.  Typing
=dbo.= and pressing =TAB= shows all tables and routines in the =dbo= schema.

*** Routine parameter completion

For stored procedures, typing =TAB= after the procedure name offers
=@parameter= names from the routine's signature.  Parameter completion is
context-aware:

- After =@param== (an already-assigned parameter), no further parameter
  names are suggested until you type a comma
- Inside string literals (single quotes), completion is suppressed

For functions, completing a function name automatically inserts parentheses
and places the cursor between them.  Eldoc shows the function signature
in the minibuffer while you type inside the parentheses.

*** TAB setup

By default, =TAB= in =sql-mode= is bound to indentation, not completion.
To make =TAB= complete when point is mid-word (and indent otherwise), add:

#+begin_src emacs-lisp
(add-hook 'sql-mode-hook
          (lambda () (setq-local tab-always-indent 'complete)))
#+end_src

** Goto definition (=M-.=)

Press =M-.= on any SQL identifier to jump to its definition:

- *Tables*: column listing (name, type, nullability, defaults)
- *Views*: the view's SQL source
- *Functions / Procedures*: full source code

Definitions open in a read-only =sql-mode= buffer.  =M-.= keeps working
inside definition buffers, so you can drill from a view into the tables it
references, or from a procedure into functions it calls.  =M-,= pops back
through the xref marker stack; =q= also goes back one level.

With no identifier at point, =M-.= prompts for a name.

The definition command is completely silent in the SQLi buffer -- no
command echo or output appears there.  Errors (object not found, etc.)
are reported in the minibuffer.

** Mode line

The mode line shows connection status:

#+begin_example
[datum:postgres:mydb:localhost:postgres]
#+end_example

Format: =[datum:dialect:database:server:user]=

** Running queries auto-refresh

When you run =:running= (or =C-c s r=), the results open in a dedicated
buffer showing session ID, user, status, duration, and SQL text for each
running query. The output uses the same tabular format as regular query
results and respects =:chars= for column width.

| Key | Action               |
|-----+----------------------|
| =g= | Manual refresh       |
| =a= | Toggle auto-refresh  |
| =q= | Quit and stop timer  |

The buffer auto-refreshes every 5 seconds by default (customizable via
=sql-datum-running-refresh-interval=). A timestamp shows the last refresh
time. The timer stops automatically when the buffer is no longer visible.

* Envelope protocol

Datum communicates structured data to Emacs via envelope lines:

#+begin_example
##DATUM:type:payload##
#+end_example

These are stripped from the visible buffer by a =comint-preoutput-filter= and
acted on silently.

| Type          | Payload              | Purpose                           |
|---------------+----------------------+-----------------------------------|
| =info=        | message              | Status message in minibuffer      |
| =warn=        | message              | Warning in minibuffer             |
| =error=       | message              | Error in minibuffer               |
| =dialect=     | mssql/postgres/mysql/sqlite/oracle/ansi | Set SQL dialect                   |
| =meta=        | key:value            | Metadata (server, database, etc.) |
| =result-file=  | path:format          | Exported file notification        |
| =definition=   | JSON {"name":...,"text":...} | DDL/source for goto-definition    |
| =introspect=   | kind:json-array      | Introspection data (replace)      |
| =introspect+=  | kind:json-array      | Introspection data (append chunk) |
| =running-text= | formatted-text       | Running queries display           |
